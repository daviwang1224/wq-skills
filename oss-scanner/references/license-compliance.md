# 开源第三方组件与开源代码管理规范

## 核心定义

### 开源第三方组件
非公司开发的、集成的开源软件组件、库、框架等。

### 开源代码
源代码公开，并遵循特定开源许可证的软件。

### 源代码自主性
评价软件自主能力的关键指标。自主率越高，风险越低。

**自主性要求**：
- 产品整体源代码自主率 ≥ 80%
- 产品核心源代码自主率 ≥ 90%

### 许可协议风险
因未遵守开源许可协议而可能导致的法律、财务或声誉损失。

## 使用方式分类

### 二进制/组件方式（推荐）
- 通过依赖管理工具引入（Maven、Gradle、npm、yarn）
- 以 .jar、.war、npm 包等形式使用
- **不影响源代码自主性**

### 源码方式（需谨慎）
- 直接将开源项目源代码复制到项目中
- 修改开源代码后集成
- **影响源代码自主性**

## 许可证风险分级

### 高风险许可证（黑名单）

以下许可证具有传染性，使用时需要特别警惕：

| 许可证 | 风险说明 | 传染性 |
|--------|----------|--------|
| **GPL (v1/v2/v3)** | 任何基于 GPL 软件的衍生产品都必须以 GPL 许可证发布，并**公开源代码** | 强传染性 |
| **AGPL (v1/v3)** | 通过网络提供服务（SaaS）时，也必须向用户提供**完整源代码**。这是最高风险的许可证之一 | 最强传染性 |
| **LGPL (v2/v2.1/v3)** | 如果修改了 LGPL 库本身，必须发布修改后的库的源代码。仅链接使用通常安全 | 中等传染性 |
| **SSPL** | 如果将使用 SSPL 的软件作为服务提供，必须公开发布该服务的所有源代码，包括管理软件、用户界面、API、自动化软件、监控软件、备份软件、存储软件和托管软件等 | 强传染性 |

### 中风险许可证

具有一些限制性条款，但不强制开源：

| 许可证 | 主要要求 |
|--------|----------|
| **MPL (Mozilla Public License)** | 修改的文件必须开源，但可以与专有代码混合 |
| **EPL (Eclipse Public License)** | 修改的模块必须开源 |
| **CDDL (Common Development and Distribution License)** | 修改的文件必须开源 |
| **CPL (Common Public License)** | 类似 EPL |

### 低风险许可证（宽松许可证）

允许商业使用，要求最少：

| 许可证 | 主要要求 |
|--------|----------|
| **MIT** | 保留版权声明和许可证声明 |
| **Apache 2.0** | 保留版权声明、许可证声明、NOTICE 文件 |
| **BSD (2-Clause/3-Clause)** | 保留版权声明 |
| **ISC** | 保留版权声明 |
| **Unlicense** | 完全公有领域，无限制 |
| **0BSD** | 类似 Unlicense |

## 常见依赖文件

### Java 后端

**Maven (pom.xml)**:
```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <version>2.7.0</version>
    </dependency>
</dependencies>
```

**Gradle (build.gradle)**:
```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web:2.7.0'
}
```

### 前端

**npm/yarn (package.json)**:
```json
{
  "dependencies": {
    "react": "^18.2.0",
    "vue": "^3.2.0"
  }
}
```

## 风险评估流程

1. **识别组件**：从依赖文件中提取组件名称和版本
2. **查询许可证**：通过 Maven Central、npm registry 或组件文档确定许可证
3. **评估使用方式**：判断是二进制方式还是源码方式
4. **风险分级**：
   - 黑名单许可证 → 高风险
   - 中等限制许可证 → 中风险
   - 宽松许可证 → 低风险
5. **自主性影响**：源码方式影响自主性，二进制方式不影响

## 处理建议

### 高风险组件
- **立即评估**：是否必须使用该组件
- **寻找替代**：查找具有宽松许可证的替代方案
- **法务咨询**：如必须使用，咨询法务部门
- **隔离使用**：通过服务化、API 等方式隔离

### 中风险组件
- **评估修改**：是否会修改组件源码
- **遵守条款**：严格遵守许可证要求
- **文档记录**：记录使用方式和合规措施

### 低风险组件
- **保留声明**：确保保留版权和许可证声明
- **定期更新**：关注安全漏洞和版本更新
