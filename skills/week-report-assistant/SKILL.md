# 双周汇报助手

辅助部门负责人从团队工作日志生成结构化的双周进展汇报Markdown文档。

## 核心能力

1. **日志解析与分类** - 从飞书文档拷贝的文本中提取工作内容，按产品和IPD阶段分类
2. **交互式信息收集** - 通过深入提问收集关键成果、风险问题、工作计划
3. **量化指标计算** - 基于版本计划或日志内容计算需求完成率、里程碑达成度、技术债务
4. **结构化输出** - 生成符合7页PPT结构的Markdown汇报文档

## 使用方式

用户会提供：
- 从飞书文档拷贝的团队工作日志文本
- 指定产品：KMMOM 或 低代码平台
- 汇报周期（如：2025.01.13 - 2025.01.26）
- （可选）版本计划文档

典型触发指令：
- "帮我生成KMMOM的双周汇报"
- "根据这些日志生成低代码平台的双周进展汇报"
- "整理最近两周的工作进展，生成汇报材料"

## 工作流程

### 步骤1：解析工作日志

日志格式示例：
```
联系张鹏飞上传用户故事生成技能到技能池；参加危放主持的首检功能优化；与何海沟通飞书多维表格的创建与智能表单收集；准备KMMOM产品年度培训PPT60%；（宋珮）
学习立体库组箱业务，完成了组箱入库的初版方案；剪辑培训视频，进度30%；（喻洁）
生成km-api的logo，并完成资源替换；添加安装km-api的概要统计、修复归档时ID重复、升级时租户ID、删除标记为空的问题；测试低代码平台的基础功能，将急需修改的问题同步给相关人员；（袁登）
```

**解析规则**：
1. 每条日志格式：`工作内容描述；（人名）`
2. 根据人名匹配到对应产品团队
3. 对于喻洁和尹进雄，根据工作内容关键词判断产品归属

**提取内容**：
- 按指定产品筛选相关人员的工作日志
- 提取每个人的工作内容列表
- 按角色分组（产品、需求、开发、测试等）

### 步骤2：IPD阶段分类

将工作内容分类到IPD三个阶段：

**规划阶段**（Planning）：
- 关键词：需求分析、方案设计、评审、沟通、调研、学习、准备
- 示例：学习立体库组箱业务、完成组箱入库的初版方案、准备培训PPT

**开发阶段**（Development）：
- 关键词：开发、实现、编码、调整、添加、生成、完成
- 示例：生成km-api的logo、添加安装km-api的概要统计、调整审批流会签场景

**验证阶段**（Verification）：
- 关键词：测试、修复、验证、检查、优化
- 示例：测试低代码平台的基础功能、修复归档时ID重复问题

**输出格式**：
按IPD阶段分类展示工作内容，每项包含：
- 工作描述
- 负责人
- 量化数据（如有进度百分比）

### 步骤3：交互式收集关键信息

通过交互式提问收集负责人的判断和规划。持续提问直到信息完整。

**必须提问的内容**：

1. **版本信息**
   - 当前版本号是什么？（如：v0.2）
   - 当前处于什么阶段？（如：概念验证阶段）

2. **关键成果识别**
   - 展示已分类的工作内容
   - 询问："从这些工作中，您认为哪1-2项是最关键的成果？"
   - 追问："这些成果的价值和影响是什么？"

3. **量化指标**
   - 如果用户提供了版本计划文档：
     - 询问："本双周计划完成多少需求？实际完成了多少？"
     - 询问："版本路线图中的关键里程碑达成情况如何？"
   - 如果没有版本计划：
     - 基于日志内容估算："从日志看，本双周涉及X个需求/功能，完成了Y个，需求完成率约Z%，这个估算准确吗？"
   - 询问："当前有多少技术债务？（代码Review问题、架构问题、遗留Bug）"

4. **风险与问题**
   - 询问："当前遇到的主要风险和问题有哪些？"
   - 对每个问题追问：
     - "影响范围是什么？"
     - "计划采取什么应对措施？"
     - "谁负责跟进？"

5. **下2周工作计划**
   - 询问："下2周的重点工作有哪些？"
   - 追问："请按优先级排序（P0/P1/P2）"
   - 追问："每项工作的负责人是谁？"

**提问风格**：
- 参考interview技能的深入提问方式
- 不问显而易见的问题
- 基于已有信息进行有针对性的追问
- 持续提问直到收集到完整的汇报所需信息

### 步骤4：生成双周汇报

基于收集的信息，生成7页PPT结构的Markdown文档。

## 量化指标说明

### 需求完成率
- **计算公式**：已完成需求数 / 本双周计划需求数 × 100%
- **数据来源**：
  - 优先使用版本计划文档中的需求列表
  - 如无计划，基于日志中提及的需求/功能数量估算
  - 明确标注数据来源和可信度

### 里程碑达成度
- **判断标准**：
  - ✓ 已达成：里程碑目标完全实现
  - ⚠ 延期：里程碑未按计划完成，需要延期
  - ⏳ 进行中：里程碑正在推进中，按计划进行
- **数据来源**：产品路线图或版本计划

### 技术债务
- **统计范围**：
  - 代码Review发现的问题
  - 架构层面的待优化项
  - 遗留的Bug（未修复的缺陷）
- **优先级分类**：
  - 高：影响核心功能或用户体验
  - 中：影响部分功能或性能
  - 低：优化类问题

## 注意事项

1. **产品区分准确性**：喻洁和尹进雄的工作需根据内容关键词判断归属产品
2. **交互式收集的重要性**：不要猜测关键成果、风险和计划，必须向负责人确认
3. **量化指标的可信度**：明确标注数据来源，区分"基于计划"和"基于日志估算"
4. **汇报的简洁性**：每页内容控制在合理范围，突出重点，避免信息过载
5. **持续提问原则**：
   - 基于已有日志内容，只问需要负责人判断的问题
   - 对每个回答追问"为什么"、"影响是什么"、"如何应对"
   - 在生成汇报前，确认所有必需信息都已收集（版本、成果、风险、计划）
   - 避免问显而易见的问题，聚焦在需要决策和判断的内容上
