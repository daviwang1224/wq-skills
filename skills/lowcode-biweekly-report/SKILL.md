# 低代码平台双周汇报助手

自动化生成低代码平台双周汇报（markdown版本，匹配PPT结构），包括数据清洗、模板生成、交互式信息收集和报告生成。

## 核心能力

1. **数据清洗与分类**：使用Python脚本处理原始txt工作日志，按人员分类工作内容
2. **模板生成**：自动生成包含完整结构的双周汇报模板（匹配PPT的6个部分），减少上下文消耗
3. **交互式信息收集**：通过问答收集关键信息（版本进度、团队成员、关键成果、风险问题、工作计划）
4. **汇报生成**：生成包含封面页、版本进度、工作进展、关键成果、工作计划、风险问题的专业汇报

## 使用方式

用户提供txt格式的工作日志，要求生成双周汇报：

```
请帮我生成低代码平台2026年2月的双周汇报，工作日志文件是 work-log-2026-02.txt
```

或者：

```
整理低代码平台最近两周的工作进展，生成双周汇报材料
```

## 工作流程

> **⚠️ 重要提示**：
> - **必须先运行清洗脚本**，不能跳过这一步
> - **不要直接读取原始txt文件**，必须通过清洗脚本生成JSON文件后再读取
> - **JSON文件只读取一次**（步骤4），后续步骤都基于已读取的内容进行
> - **填充模板后立即清除所有注释**：每个部分填充完成后，立即删除该部分的所有 `<!-- -->` 注释
> - 清洗脚本会自动完成数据清洗和按人员分类

### 步骤1：日志清洗（必须先执行）

运行清洗脚本处理原始txt工作日志，输出JSON格式的清洗后数据：

```bash
python scripts/clean_work_log.py <input.txt> <output.json>
```

脚本输出JSON文件（`[yyyy-mm]_工作日志.json`），包含summary（统计）、details（按人员分类的工作内容）。

### 步骤2：生成汇报模板

运行init_report.py生成包含完整结构的双周汇报模板（匹配PPT结构）：

```bash
python scripts/init_report.py <input.json> <output.md>
```

脚本会生成包含6个部分的Markdown模板（`低代码平台双周汇报_[yyyy-mm].md`），使用TODO标记指示需要填充的部分。这样可以避免从零开始写整个文件，减少上下文消耗。

**模板结构**（对应PPT的6张幻灯片）：
1. 封面页：标题、汇报周期、主讲人、日期、公司信息
2. 版本进度总览：版本号、阶段、目标发布时间、版本定位、里程碑达成度、团队成员
3. 当前工作进展：序号+完成事项+负责人+说明的表格
4. 关键成果：1-2项重点成果，包含价值、详情、负责人
5. 下2周工作计划：重点目标+优先级任务表格
6. 风险与问题：问题描述+影响+应对措施+责任人

### 步骤3：读取团队结构

**强制读取** [references/team-structure.md](references/team-structure.md)，了解低代码平台团队成员和岗位信息。

**目的**：了解团队成员的岗位（产品经理、架构师、前端/后端开发等），便于后续分析和填充。

### 步骤4：读取JSON文件

**一次性读取**清洗后的JSON文件，加载所有数据到上下文：
- `summary`：周期、人员数、工作日数等统计信息
- `details`：每个成员的具体工作内容（按日期分类）

**⚠️ 重要**：此步骤后，后续步骤都基于已读取的JSON内容进行，**禁止重复读取JSON文件**。

### 步骤5：交互式收集关键信息

通过交互式提问收集负责人的判断和规划，持续提问直到信息完整。

**必须提问的6类信息**：

1. **版本信息**：版本号、阶段、目标发布时间、版本定位
2. **里程碑达成度**：关键里程碑的达成情况（✓ 已达成 / ⚠ 延期 / ⏳ 进行中）及具体内容
3. **团队成员信息**：总人数、全职/兼职人数、测试负责人、跨团队成员
4. **关键成果识别**：展示JSON数据中的工作内容，询问哪1-2项是最关键成果，追问价值、影响和具体完成内容
5. **风险与问题**：主要风险和问题、影响范围、应对措施（可多条）、责任人
6. **下2周工作计划**：重点目标（一句话）、重点工作、优先级（P0/P1/P2）、负责人、预期产出

**提问原则**：基于已有信息有针对性追问，不问显而易见的问题。

### 步骤6：填充版本进度总览

基于交互收集的信息，填充模板的"一、版本进度总览"部分。

**填充内容**：
- 版本号、阶段、目标发布时间
- 版本定位
- 里程碑达成度表格（指标 | 内容 | 说明）
- 团队成员信息（总人数、全职/兼职人数、角色分工、跨团队成员）

**输出要求**：参考 [references/output-patterns.md](references/output-patterns.md) "版本进度总览原则"

### 步骤7：填充当前工作进展

基于已读取的JSON数据，填充模板的"二、当前工作进展"部分。

**工作内容筛选**：
- 从JSON数据中选择8-12项重点工作
- 按序号列出：序号 | 完成事项 | 负责人 | 说明
- 完成事项：控制在20字以内（标题性质）
- 说明：详细描述工作内容，控制在100字以内

**输出要求**：参考 [references/output-patterns.md](references/output-patterns.md) "当前工作进展原则"

### 步骤8：填充关键成果

基于交互收集的关键成果信息和JSON数据，填充模板的"三、关键成果"部分。

**填充内容**（每个成果）：
- 成果标题
- 价值：说明成果的业务价值和战略意义
- 详情：使用列表形式，列出3-5个具体的完成内容
- 负责人：列出主要贡献者

**输出要求**：参考 [references/output-patterns.md](references/output-patterns.md) "关键成果原则"

### 步骤9：填充下2周工作计划

基于交互收集的工作计划信息，填充模板的"四、下2周工作计划"部分。

**填充内容**：
- 重点目标：用一句话说明下2周的核心目标
- 任务表格：优先级 | 任务 | 负责人 | 预期产出
- 任务数量：4-6项，P0任务3-4项，P1任务1-2项

**输出要求**：参考 [references/output-patterns.md](references/output-patterns.md) "下2周工作计划原则"

### 步骤10：填充风险与问题

基于交互收集的风险问题信息，填充模板的"五、风险与问题"部分。

**填充内容**：
- 问题描述：控制在20字以内
- 影响：控制在30字以内
- 应对措施：使用编号列表，每项控制在40字以内
- 责任人：明确负责人

**输出要求**：参考 [references/output-patterns.md](references/output-patterns.md) "风险与问题原则"

### 步骤11：质量检查

完成报告生成后，必须执行质量检查，确保报告满足所有要求。

**检查流程**：

1. **读取生成的报告**：读取生成的双周汇报
2. **读取质量检查清单**：参考 [references/output-patterns.md](references/output-patterns.md) "质量检查清单"部分
3. **逐项检查**：按照检查清单的5个类别（结构完整性、数据准确性、内容质量、格式规范性、语言专业性）逐项验证
4. **修复问题**：如果发现问题，直接修复
5. **输出检查结果**：向用户报告检查结果，说明是否通过检查，如有问题已修复

**检查要点**：

- **结构完整性**：封面页信息齐全，五个主体部分齐全，标题层级正确，注释已清除
- **数据准确性**：版本号、阶段、目标发布时间准确，团队成员信息准确
- **内容质量**：关键成果突出（1-2项），工作计划有重点目标，风险问题有应对措施，工作进展选择8-12项重点工作
- **格式规范性**：表格格式正确，日期格式统一，优先级标记统一，里程碑状态标记统一
- **语言专业性**：语言正式、客观、专业，无口语化表达，无错别字

**⚠️ 重要**：质量检查是强制步骤，不可跳过。如果发现问题，必须修复后再向用户交付报告。
