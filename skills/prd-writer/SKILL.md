# PRD Writer

## Core Rule

在完成第一阶段诊断、页面结构确认和第二阶段概念对齐之前，不要输出完整的 PRD、功能列表或结构化需求文档。在概念版 PRD 确认之前，保持对话交互式推进。
默认使用中文，除非用户要求其他语言。

## Workflow

### Phase 1: 发现与页面结构

在起草之前，从用户、业务、技术和页面结构四个维度收集必要信息。每轮最多问 3 个问题，然后等待用户回复。如果用户已经提供过答案，不要重复提问。

用户维度:
- 目标用户是谁？尽可能包含年龄范围、角色/背景和使用场景。
-  他们今天如何解决这个问题？
-  产品应该帮助用户完成的具体任务是什么？
-  使用频率：每天、每周、偶尔，还是其他

业务维度:
- 产品是免费还是付费？
- 如果付费，采用哪种模式：一次性购买、订阅、免费增值，还是其他？
-  这是一个验证项目还是一个长期产品？

技术维度:
- 产品形态：小程序、iOS、Android、H5、Web App，还是其他平台？
-  是否需要账号和登录？
-  数据存储在哪里：本地存储还是云端数据库？
-  是否依赖第三方服务，如地图、支付、AI API、通知或分析？
-  这个版本预计需要多少周完成？

页面结构维度:
- 主要页面或 Tab 有哪些？例如：首页、日历页、记录页、统计页、个人中心。
- 每个 MVP 功能映射到哪个页面：入口在哪里，结果出现在哪里？
- 产品应使用哪种导航模式：底部 Tab、侧边导航、顶部导航、单页流程还是向导？
- 是否需要辅助页面如引导页、登录页、设置页、详情页、编辑页、订阅/支付页或帮助页？

当答案模糊时，提出最少量的追问以消除歧义。一旦拥有足够的信息来编写精确的概念版 PRD，不要过度追问。

如果用户不知道页面结构，基于 MVP 功能提出一个最简页面结构草案并要求确认。在用户提供或确认页面结构之前，不要进入概念版 PRD。

### Phase 2: 概念版 PRD 对齐

第一阶段完成后，立即输出以下概念版 PRD 模板。内容控制在 200 个中文字符以内，不要增加额外章节。

```markdown
## 概念版 PRD

核心用户：[一句话描述目标用户]
要解决的一件事：[一个具体痛点，不能写多个]
产品形态：[平台]
页面结构：[主页面/Tab，不超过 5 个]
最小可用版功能（≤ 3 条）：
  1. [功能一]
  2. [功能二]
  3. [功能三]
本版本不做：[明确排除的功能]
商业模式：[收费方式 或 免费]
技术前提：[账号体系 / 数据方案 / 第三方依赖]
```

然后准确提问：

```markdown
以上是概念版 PRD，请确认：① 方向是否对齐？② 有需要调整的地方吗？
确认后进入落地版。
```

如果用户要求修改，修订概念版 PRD 并再次请求确认。在用户确认之前，不要进入第三阶段。

### Phase 3: 范围冻结

在编写完整 PRD 之前，要求用户确认冻结范围：

```markdown
在开始落地版之前，请确认以下内容不会再修改：

✅ 目标用户：[从概念版填入]
✅ 核心功能（≤ 3 条）：[从概念版填入]
✅ 平台选择：[从概念版填入]
✅ 页面结构：[从概念版填入]
✅ 本版本不做：[从概念版填入]

请回复"确认"后继续。
```

如果用户改变了范围，回到第二阶段重新对齐。只有在用户明确确认后才继续。

### Phase 4: 落地版 PRD

在编写完整 PRD 之前，加载 `references/prd-template.md`。严格遵循其结构。

规则:
- Do n不要使用模糊占位符，如"待定"、"TBD"、"视情况而定"或"后续补充"。
- 每个章节都必须填写具体决策。如果信息缺失，推断一个保守的默认值并标注为"建议默认值："，而不是留空。
- 覆盖每个 P0/P1 功能的用户流程、状态机、输入字段规则、文案规则和异常处理。
- 每个用户流程中必须包含正常路径和异常路径。
- 除非用户提供其他日期，否则使用当前日期作为文档日期。
- 未指定作者时，以用户为作者。

在最终定稿前，执行 `references/prd-template.md` 中的自检清单。如果发现任何盲点，在呈现最终 PRD 之前补充完整。

## 输出行为

当用户要求可下载文件或 .docx 格式时，将此技能与适当的文档生成技能结合使用。否则，以 Markdown 格式输出 PRD。

如果用户坚持"直接写完整 PRD"而不经过诊断，简要说明该技能首先需要几个对齐问题以避免产出不可用的 PRD，然后开始第一阶段，提出最多 3 个问题。
