# 开源组件扫描与合规分析

## 核心能力

- 扫描 Java 后端项目（Maven pom.xml、Gradle build.gradle）
- 扫描前端项目（React/Vue 的 package.json）
- 识别开源组件的许可证类型
- 评估许可证风险等级（高/中/低）
- 判断是否影响源代码自主性
- 生成 Markdown 格式的合规分析报告

## 使用方式

用户可以通过以下方式触发此技能：

- "扫描这个项目的开源组件"
- "检查项目的许可证合规性"
- "分析开源依赖的风险"
- "生成开源软件使用报告"
- "检查是否有 GPL 许可证的组件"

## 工作流程

### 第一步：识别项目类型并确认扫描范围

1. **自动搜索依赖文件**：
   ```
   **/pom.xml          # Maven
   **/build.gradle*    # Gradle
   **/package.json     # npm/yarn
   ```

2. **确认扫描范围**：
   - **默认行为**：扫描所有发现的依赖文件（包括子模块）
   - **仅在以下情况询问用户**：
     - 发现多种项目类型（Java + 前端）→ 询问是否全部扫描
     - 发现前端项目 → 询问是否包含 devDependencies
   - 如果只有单一项目类型，直接开始扫描，无需询问

3. **处理能力**：
   - 搜索依赖文件
   - 读取依赖文件内容
   - 查询许可证信息
   - 在用户请求时导出报告

### 第二步：扫描依赖文件

根据项目类型，读取相应的依赖文件：

**Java Maven 项目**：
- 读取 `pom.xml`
- 提取 `<dependencies>` 部分的所有依赖
- 记录 groupId、artifactId、version

**Java Gradle 项目**：
- 读取 `build.gradle` 或 `build.gradle.kts`
- 提取 `dependencies` 块中的依赖声明
- 解析 implementation、api、compileOnly 等配置

**前端项目**：
- 读取 `package.json`
- 提取 `dependencies` 和 `devDependencies`
- 记录包名和版本

### 第三步：识别许可证信息

**对每个组件执行以下步骤**：

1. **查询许可证**：
   - **Java 组件**：搜索 `Maven {groupId}:{artifactId} license`
   - **npm 组件**：搜索 `npm {package-name} license`
   - **优先来源**：Maven Central、npmjs.com 官方页面

2. **标准化许可证名称**：
   - `Apache License 2.0` → `Apache-2.0`
   - `MIT License` → `MIT`
   - `GNU General Public License v3.0` → `GPL-3.0`
   - 多许可证：保留原格式（如 `MIT OR Apache-2.0`）

3. **处理未知许可证**：
   - 如果无法确定许可证，标记为 `Unknown`
   - 在报告中单独列出，建议人工核查

### 第四步：风险评估

参考 [license-compliance.md](references/license-compliance.md) 中的风险分级标准。

**1. 判断使用方式**（决策规则）：

| 来源 | 使用方式 |
|------|----------|
| pom.xml / build.gradle / package.json 中的依赖 | 二进制/组件 |
| 项目 src/ 目录下的第三方代码文件 | 源码方式 |

**默认**：依赖文件中的组件均为"二进制/组件方式"

**2. 评估许可协议风险**（查表）：

| 许可证类型 | 风险等级 | 示例 |
|------------|----------|------|
| GPL, AGPL, LGPL, SSPL | **高风险** | GPL-3.0, AGPL-3.0 |
| MPL, EPL, CDDL, CPL | 中风险 | MPL-2.0, EPL-1.0 |
| MIT, Apache, BSD, ISC | 低风险 | MIT, Apache-2.0 |
| Unknown | **高风险** | 未知许可证 |

**3. 判断是否影响源代码自主性**（规则）：

| 使用方式 | 是否影响自主性 |
|----------|----------------|
| 源码方式 | **是** |
| 二进制/组件方式 | 否 |

### 第五步：生成分析报告

生成 Markdown 格式的报告，包含以下内容：

```markdown
# 开源软件使用情况分析报告

## 项目信息
- **项目路径**：[路径]
- **扫描时间**：[时间]
- **项目类型**：[Java Maven / Java Gradle / 前端 React/Vue]

## 依赖统计
- **总依赖数**：[数量]
- **高风险组件**：[数量]
- **中风险组件**：[数量]
- **低风险组件**：[数量]

## 详细分析

| 开源组件/代码 (版本) | 使用方式 | 开源许可证 | 许可协议风险 | 是否影响源代码自主性 |
| :--- | :--- | :--- | :--- | :--- |
| spring-boot-starter-web v2.7.0 | 二进制/组件 | Apache-2.0 | 低风险 | 否 |
| fastjson v1.2.83 | 二进制/组件 | Apache-2.0 | 低风险 | 否 |
| some-gpl-lib v1.0.0 | 二进制/组件 | GPL-3.0 | **高风险** | 否 |
| custom-code vN/A | 源码方式 | MIT | 低风险 | **是** |

**表格说明**：
- 所有列使用左对齐 (`:---`)
- 高风险许可证用 `**高风险**` 加粗标注
- 影响源代码自主性的用 `**是**` 加粗标注
- 版本号格式：`v{version}` 或 `vN/A`（源码方式无版本）

## 高风险组件汇总

[如果存在高风险组件，列出详细信息和处理建议]

### 组件：[组件名]
- **许可证**：[许可证名称]
- **风险说明**：[传染性说明]
- **处理建议**：
  1. 评估是否必须使用该组件
  2. 寻找具有宽松许可证的替代方案
  3. 如必须使用，咨询法务部门
  4. 考虑通过服务化、API 等方式隔离

## 合规建议

[根据扫描结果提供整体合规建议]
```

### 第六步：后续交互

**报告生成后，主动询问**：

1. "是否需要将报告导出到文件？（默认文件名：`oss-analysis-report-{timestamp}.md`）"

**如果存在高风险组件，额外询问**：

2. "发现 [N] 个高风险组件，是否需要针对某个组件进行详细分析？"
3. "是否需要搜索高风险组件的替代方案？"

**交互原则**：
- 一次最多询问 3 个问题
- 如果用户未响应，默认不导出文件
- 如果用户选择详细分析，加载 [license-compliance.md](references/license-compliance.md) 提供深度建议

## 注意事项

1. **许可证识别准确性**：
   - 许可证信息可能随版本变化
   - 某些组件可能有多个许可证选项
   - 建议交叉验证多个来源

2. **风险评估局限性**：
   - 本工具提供初步风险评估
   - 高风险组件需要法务部门最终确认
   - 实际风险取决于具体使用方式

3. **源代码自主性计算**：
   - 需要结合项目总代码行数计算自主率
   - 本工具仅标识是否影响自主性
   - 详细计算需要额外工具支持

4. **持续监控**：
   - 依赖更新可能改变许可证
   - 建议定期重新扫描
   - 关注安全漏洞和版本更新
