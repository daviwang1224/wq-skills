# 需求澄清访谈

## 核心定位

将模糊、口语化、零散的需求澄清为可执行的任务定义，帮助后续执行者明确背景、目标、范围、任务、约束和待确认问题。

## 工作流程

### 1. 判断是否需要澄清

当用户需求存在以下任一问题时，进入访谈：

- 目标不明确：不知道最终要解决什么问题或达成什么结果
- 背景不足：缺少业务背景、使用场景、目标对象或已有材料
- 范围不清：不知道做什么、不做什么、边界在哪里
- 任务不实：只有愿望或方向，缺少可执行动作
- 约束缺失：缺少时间、格式、风格、技术、资源、质量或合规要求
- 交付不明：不知道最终需要文档、方案、代码、清单、报告还是其他产物

### 2. 提问原则

- 先问会影响方向的问题，再问细节问题。
- 每轮最多问 3 个关键问题。
- 优先使用选择题或带默认建议的问题，降低用户回答成本。
- 不确定就明确说“不知道”，不要替用户编造事实。
- 可以给出建议默认值，但必须标明这是“建议”，不能当作已确认信息。

### 3. 访谈维度

围绕以下维度收集信息：

- 背景：为什么要做，当前处境是什么，已有信息有哪些
- 目标：希望解决什么问题，达成什么结果
- 对象：给谁用，影响谁，谁验收
- 范围：包含哪些内容，不包含哪些内容
- 任务：具体要完成哪些动作
- 交付：最终输出什么形式的成果
- 约束：时间、格式、风格、资源、技术、质量、合规等限制
- 标准：怎样判断完成，怎样判断质量达标

### 4. 整理规则

访谈结束后，按“已确认”和“待确认”区分信息。

- 已确认信息：用户明确说过，或可从用户提供材料直接得出
- 建议假设：为了推进任务提出的合理默认值，必须标注为建议
- 待确认问题：只保留会影响执行方向、范围、交付或质量的问题
- 冲突信息：直接指出冲突，不做美化

## 输出结构

访谈完成后，使用以下结构输出：

```markdown
## 1. 需求结论
[一句话说明最终要解决的问题和期望结果。]

## 2. 背景目标
[说明为什么要做、给谁用、希望产生什么价值。区分已确认信息和建议假设。]

## 3. 范围边界
[明确包含什么、不包含什么、涉及哪些对象、场景或模块。]

## 4. 任务定义
[拆成可执行任务，说明要做的具体动作和最终交付物。]

## 5. 约束与标准
[列出格式、风格、时间、资源、技术、质量、合规等限制，以及完成标准。]

## 6. 待确认问题
[只列仍会影响执行的关键问题，并给出建议下一步。]
```

## 示例

用户输入：

> 帮我整理一下团队周报，要专业一点。

优先提问：

1. 这份周报给谁看：直属领导、跨部门评审，还是团队内部同步？
2. 周报要覆盖哪个时间范围，以及已有原始材料在哪里？
3. 最终交付格式要 Markdown、Word，还是直接生成可粘贴到飞书的文本？

访谈后的输出：

```markdown
## 1. 需求结论
整理一份面向直属领导的团队周报，用于汇报本周进展、问题和下周计划。

## 2. 背景目标
已确认：周报用于管理层同步，重点展示团队工作进度和风险。
建议假设：表达应简洁、事实优先，避免泛泛总结。

## 3. 范围边界
包含：本周完成事项、进行中事项、风险问题、下周计划。
不包含：个人绩效评价、详细过程记录、未确认的成果夸大。

## 4. 任务定义
基于用户提供的原始工作记录，归类整理团队周报，并输出可直接粘贴使用的文本。

## 5. 约束与标准
格式：Markdown。
风格：清晰、务实、适合管理层快速阅读。
标准：事项归类准确，风险表达具体，下周计划可执行。

## 6. 待确认问题
请补充原始工作记录和周报时间范围；没有这两项，无法可靠生成正文。
```
