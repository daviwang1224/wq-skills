# 文章结构模式

## 使用原则

- 本文件只提供写作路径，不定义文章分类。
- 一次写作只选一个主路径，必要时局部借用另一条路径；不要把多种路径并列堆进正文。
- 路径只用于组织作者思路，正文不说明“本文采用某某结构”。
- 结论必须明确，但不强制结论先行；开头可以先写场景、疑问、观察、案例或外部观点。
- 固定的是推进方式，不是段落模板。可以按素材调整顺序，但不能把文章写散。

## 观察推进型

适合素材来自日常工作现象、组织现象、产品现象、协作现象或管理观察。

推荐路径：

1. 写一个具体观察或现象。
2. 说明为什么这个现象值得注意。
3. 推出背后的逻辑、问题意识或认知差异。
4. 迁移到工作、产品、团队或组织场景。
5. 用一个清楚判断或一个继续思考的问题收束。

注意事项：

- 观察要具体，不要一上来抽象概括。
- 不要强行把现象写成批评或问题诊断。

## 疑问展开型

适合素材里有困惑、反常感、争议点，或用户暂时只有一个说不清的判断雏形。

推荐路径：

1. 提出一个真实疑问。
2. 说明常见理解为什么不够。
3. 逐层分析疑问背后的逻辑。
4. 形成新的理解或判断。
5. 收束为读者可带走的认知点。

注意事项：

- 疑问要真实，不要制造悬念。
- 分析过程要推进判断，不能绕圈。

## 案例复盘型

适合素材来自一次具体事件、项目、会议、协作过程或管理动作。

推荐路径：

1. 交代一个足够具体的事件或场景。
2. 找出事件中的关键变化、冲突或转折。
3. 抽象出可迁移的经验、方法或判断。
4. 说明类似场景下应该如何理解。
5. 用一句方法口径或判断收束。

注意事项：

- 案例只服务观点，不要写成流水账。
- 不把个案扩大成组织共识、制度结论或对具体人的批评。

## 理论转化型

适合用户提供外部文章、书籍、理论、课程、访谈或公共案例，并希望转成工作认知。

推荐路径：

1. 简述外部观点或理论，不堆背景。
2. 写出作者自己的理解，而不是复述原文。
3. 映射到实际工作、产品、团队或组织场景。
4. 提炼新的判断、提醒或方法。
5. 说明适用边界，避免理论泛化。

注意事项：

- 外部事实必须有来源；不知道来源就说明不知道。
- 理论只负责帮助理解素材，不能压过工作判断。

## 判断论证型

适合用户已经给出明确判断，文章重点是把判断讲清楚、讲可信。

推荐路径：

1. 提出核心判断。
2. 说明为什么需要这个判断。
3. 展开判断成立的关键理由。
4. 明确适用边界和常见误区。
5. 用一句简明口径收束。

注意事项：

- 这个路径可以结论先行，但不是默认路径。
- 判断要有素材支撑，不能把个人看法写成组织定论。

## 方法沉淀型

适合用户想把一类工作经验、协作方式、产品方法或 AI 使用方法沉淀下来。

推荐路径：

1. 从具体工作场景或难点进入。
2. 说明旧做法为什么不够。
3. 提炼方法、原则或操作规则。
4. 说明适用边界和不该怎么做。
5. 用便于复用的口径收束。

注意事项：

- 方法必须来自素材支撑，不能凭空扩展成制度。
- 边界比口号重要：说明什么情况下适用，什么情况下不适用。

## 路径选择规则

- 用户素材来自现象、观察或管理感受，优先选观察推进型。
- 用户素材来自困惑、争议或反常现象，优先选疑问展开型。
- 用户素材来自具体事件或项目经历，优先选案例复盘型。
- 用户素材来自外部文章、理论或书籍，优先选理论转化型。
- 用户已经给出明确判断，优先选判断论证型。
- 用户希望沉淀做法、原则或工作口径，优先选方法沉淀型。
- 如果两条路径都成立，以用户的使用场景决定；仍无法判断时，最多给用户 2 个写法方向。
