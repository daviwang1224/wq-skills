# 统一语言（Ubiquitous Language）

主动维护项目统一语言。重点是挑战模糊说法、识别术语冲突、压实领域边界，并在术语确认后立即更新 `CONTEXT.md`。

只在需要改变或维护领域语言时使用本技能。单纯读取 `CONTEXT.md` 理解词汇，不算使用本技能。

## 文件结构

多数仓库只有一个上下文：

```text
/
├── CONTEXT.md
└── src/
```

如果根目录存在 `CONTEXT-MAP.md`，说明仓库有多个上下文：

```text
/
├── CONTEXT-MAP.md
└── src/
    ├── ordering/
    │   └── CONTEXT.md
    └── billing/
        └── CONTEXT.md
```

按需创建文件：没有术语要沉淀时，不创建 `CONTEXT.md`；第一次确认术语后再创建。

## 工作流程

### 检查现有上下文

先检查仓库根目录是否存在 `CONTEXT-MAP.md` 或 `CONTEXT.md`：

- 有 `CONTEXT-MAP.md`：读取它，确认当前讨论属于哪个上下文。
- 只有根目录 `CONTEXT.md`：按单一上下文处理。
- 两者都没有：先继续澄清概念，等第一个术语确认后再创建根目录 `CONTEXT.md`。

如果存在多个上下文，但当前话题归属不清，必须询问用户，不能自行猜测。

### 挑战术语冲突

当用户使用的词和 `CONTEXT.md` 中已有定义冲突时，立即指出：

```text
当前术语表把“取消”定义为 X，但你现在表达的是 Y。这里到底应该采用哪个含义？
```

不要把两个冲突含义都写进去。统一语言必须收敛，不是罗列口径。

### 收敛模糊语言

当用户使用含糊、宽泛或重载词时，提出更精确的候选术语：

```text
你说的“账号”不够精确。这里是指登录用户 User，还是签约客户 Customer？
```

每次只处理当前讨论中影响理解的术语，不做大而全的词典工程。

### 用场景压实边界

当领域关系不清时，用具体场景测试边界：

- 这个动作发生在什么对象上？
- 是否允许部分处理？
- 谁拥有这个概念？
- 其他上下文只能引用 ID，还是可以持有完整信息？
- 历史状态是否保留？

场景必须服务于术语澄清，不要扩展成需求设计。

### 和代码交叉核对

当用户说明业务规则或领域关系时，检查代码是否一致。发现矛盾时直接指出：

```text
代码里取消的是整张订单，但你刚才说支持取消订单行。这里文档和代码冲突，需要先确认真实规则。
```

只基于可读取到的代码和文档下结论；证据不足时明确说“不知道”。

### 更新 CONTEXT.md

术语一旦确认，立即更新对应 `CONTEXT.md`。不要攒到最后批量处理。

`CONTEXT.md` 只记录领域语言：

- 写领域术语，不写实现细节。
- 写概念定义，不写流程规格。
- 写项目特有概念，不写通用编程概念。
- 写推荐术语，并列出应避免的同义词或错误说法。

格式遵循 [context-format.md](references/context-format.md)。
